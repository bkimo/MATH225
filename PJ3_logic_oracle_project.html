Discrete Thinking — Chapter Project

The Logic Oracle

A neuro-symbolic system for formal policy verification using BDDs and LLMs

1. Project Overview

Natural language is expressive but imprecise. Business rules and policy documents are full of subtle logical dependencies — "You may proceed if the request is approved and the budget is not frozen, unless the requester holds director-level clearance." A human reads this and nods. A formal system breaks it into irrefutable pieces.

The Logic Oracle bridges these two worlds using a neuro-symbolic architecture: a Large Language Model (LLM) handles the imprecision and ambiguity of prose, while a Reduced Ordered Binary Decision Diagram (ROBDD) handles the actual logical computation with mathematical certainty. Once a rule is encoded in the BDD, the graph never makes a mistake.

Key insight

An LLM is excellent at extracting intent from language. A BDD is excellent at verifying logical consequences. Neither alone solves the problem. Together, they form a system that is both interpretable and rigorous.

Architecture overview — neuro-symbolic pipeline

Core goals

Students will build a system that accomplishes four things:

  • Symbolic encoding — extract Boolean variables and operators from prose using an LLM
  • Recursive construction — build a DAG via Shannon expansion, where each internal node is a decision point
  • Formal verification — detect contradictions (a variable simultaneously forced to True and False) and dead states
  • Traceable explanation — translate graph paths back into natural language via a second LLM call

2. Technical Architecture

2.1 Shannon expansion — the recursive engine

Every BDD is built by recursively applying the Shannon expansion. For a Boolean function f and a chosen variable xi:

Shannon Expansion

f = (xi ∧ f|xi=1) ∨ (¬xi ∧ f|xi=0)

The function f is broken down by "fixing" one variable at a time — high branch when the variable is True, low branch when False — until reaching the terminal nodes 0 (False) or 1 (True). This is inherently recursive, and its termination is guaranteed.

2.2 The reduction step — from tree to DAG

After recursive expansion, many sub-trees are identical. The ROBDD optimiser identifies and merges these redundant sub-graphs. The result is a canonical Directed Acyclic Graph — compact, deterministic, and unique for a given variable ordering and expression.

Canonicity — the power of ROBDDs

Two logically equivalent expressions always produce the same ROBDD under the same variable ordering. This is what makes formal verification possible: equality of graphs proves logical equivalence. No need to enumerate rows.

2.3 Variable ordering — the hidden complexity

The compactness of a BDD depends critically on the order in which variables are evaluated. A poor ordering can cause exponential blowup; an optimal ordering can keep the graph linear in the number of variables.

Effect of variable ordering on BDD size — same expression, two orderings
Good ordering: x₁ → x₂ → x₃
Poor ordering: x₃ → x₁ → x₂
Key takeaway for Week 1

Variable ordering determines BDD size. A poor ordering can cause exponential blowup — the BDD for a multiplier circuit is provably exponential under any ordering. Students should experiment with at least three orderings and document the node-count differences.

Week 1 — Symbolic Core (detail)

Students focus on building "ground truth" before touching an LLM. The central task is understanding that a Boolean expression has a unique ROBDD representation under a fixed variable ordering, and that this representation can be traversed to produce a complete truth table. 

# Week 1 starter — BDD truth table traversal
from pyeda.inter import exprvars, expr

# Define variables and expression
A, B, C = exprvars('A', 3)
# Expression: (A AND NOT B) OR C
f = (A[0] & ~A[1]) | A[2]

# Convert to BDD and print truth table
bdd = f.to_bdd()
for point, val in f.iter_points():
    print(point, "→", int(f.restrict(point)))

# Try different orderings — observe node count change

Week 1 research task

For expressions with 3, 4, 5, and 6 variables, compare the BDD node count with the truth table row count (2n). Plot the compression ratio. This directly motivates the value of the BDD representation.

Week 2 — Neural Bridge (detail)

Students build the LLM parser and — critically — measure its reliability against a fixed test suite. The most common failure modes are operator precedence errors (misplacing parentheses), scope ambiguity (does "or not" mean "(or not)" or "or (not)"), and hallucinated variables (the LLM invents a variable not present in the prose).

Common failure mode — hallucinated variables

If the LLM outputs a variable name that does not appear in the defined variable list, the BDD builder will raise an error. Students must implement graceful error handling: re-prompt the LLM, ask for clarification, or fall back to a default. Document every failure case in the test suite.

Week 3 — Verification & Explanation (detail)

The recursive path-finder must handle an important design decision: a BDD terminal node 0 may be reachable via many different variable assignments. Students must explicitly choose and justify one of three strategies:

  • All paths — enumerate every variable assignment leading to the 0 terminal. Useful for audit trails but expensive for large BDDs.
  • Shortest path — find the fewest-variable-assignments path via BFS. Good for minimal explanations.
  • Representative path — pick one path deterministically (e.g., always take the True branch when both are available). Simplest to implement; useful for initial explanation.

The explanation loop feeds the chosen path back to the LLM using a structured prompt (see §4 for the full prompt template). The LLM translates the path into an English sentence that references the original variable descriptions.

4. The Logic-to-JSON Prompt (Appendix Template)

This is the core engineering artefact of Week 2. The system prompt must be precise enough to constrain the LLM's output to a clean, parseable JSON object, while flexible enough to handle varied prose inputs. The following template has been designed to handle the most common failure modes — include it in your appendix verbatim.

4.1 System prompt (paste into your LLM client)

You are a formal logic extraction engine. Your ONLY job is to convert
natural language policy rules into a structured JSON object. You must
NEVER explain your reasoning or add commentary — output ONLY valid JSON.

=== OUTPUT FORMAT ===
{
  "variables": {
    "A": "<short noun phrase describing what A means>",
    "B": "<short noun phrase describing what B means>",
    ...
  },
  "expression": "<Boolean expression using only the variable names above>",
  "notes": "<optional: flag any ambiguity in the source text>"
}

=== RULES FOR THE EXPRESSION FIELD ===
- Use ONLY variables defined in the "variables" field. Never invent new ones.
- Operators: AND (&), OR (|), NOT (~), implies (->), equivalence (<->)
- Use parentheses explicitly to resolve all precedence ambiguity.
- If a natural language statement is ambiguous, choose the most conservative
  interpretation and flag it in the "notes" field.
- Never output partial expressions. If you cannot fully encode the input,
  set "expression" to null and explain in "notes".

=== EXAMPLES ===
Input: "A package ships if it is paid for and not oversized."
Output:
{
  "variables": {
    "P": "Package is paid for",
    "O": "Package is oversized"
  },
  "expression": "P & ~O",
  "notes": null
}

Input: "Access is granted if the user is verified or is an admin,
        but only when the system is online."
Output:
{
  "variables": {
    "V": "User is verified",
    "D": "User is an admin",
    "S": "System is online"
  },
  "expression": "(V | D) & S",
  "notes": null
}

Input: "You may enter unless you are banned and not on the guest list,
        or if the event is public."
Output:
{
  "variables": {
    "B": "User is banned",
    "G": "User is on the guest list",
    "E": "Event is public"
  },
  "expression": "(~(B & ~G)) | E",
  "notes": "Ambiguity: 'unless...or' could bind differently. Chose conservative
            reading: denial condition is (B AND NOT G), which is negated."
}

5. Research Questions

To elevate this from a coding assignment to undergraduate research, students should document findings for at least two of the following questions.

01

Logic depth vs. hallucination

At what nesting depth does the LLM begin to fail at symbolic translation? Design inputs with 1, 2, 3, and 4 levels of parenthesis nesting. Measure accuracy against the test suite. Plot a hallucination-rate curve. This is an open empirical question — publishable at the undergraduate level with a clean dataset and methodology.

02

Graph compression vs. truth table size

As the number of variables n grows, compare BDD node count with truth table row count (2n). For which classes of expressions is compression greatest? For which is it worst? (Hint: parity functions have exponential BDDs under any ordering.)

03

Dead code detection

A "dead" logical state is a variable assignment that can never occur — either because it is a logical contradiction (A is simultaneously True and False) or because earlier rules make it unreachable. How can BDD structure reveal dead states without enumerating all 2n rows? Can you formalise this as a graph property?

04

Multi-rule conflict detection

Encode two policy rules separately as BDDs, then compute their conjunction. If the result is the constant-0 function (the BDD collapses to a single False terminal), the rules are contradictory. Implement this check and test it on at least three pairs of conflicting real-world policies.

6. Suggested Python Stack

Logic & BDD

pyeda dd sympy.logic

pyeda is the most comprehensive. dd is lighter and faster for large BDDs. sympy.logic is useful for symbolic simplification before BDD construction.

LLM integration

anthropic openai langchain

Use the anthropic or openai SDK directly for Week 2 — keep the LLM call transparent. langchain is useful in Week 3 for chaining the two prompt calls.

Visualisation

graphviz networkx matplotlib

graphviz renders the BDD as a proper DAG. networkx + matplotlib are useful for the research question plots.

Interface

streamlit click rich

streamlit for the web app deliverable. click + rich for a polished CLI — both are acceptable for the Week 3 deliverable.

Grading note for instructors

The core grading criterion is not correctness of the LLM output — it is the rigour of the symbolic layer. A system where the BDD is correct and the LLM makes occasional errors (properly caught and handled) scores higher than one where the LLM appears to work but the BDD construction is bypassed or trivialised.
